## 引言

前段时间刷 X，刷到了 Monad Builder Camp 的暑期实习邀请。说实话，当时我脑子里对 Web3 的概念还停留在“听过但没真正碰过”的阶段——大概就像很多人听说 Rust 很火但从来没用过一样。在详细了解 Monad Builder Camp 之后，我只是犹豫了那么一瞬间，就选择了报名。

结果居然过了筛选。
![](/images/blog/07/image-1.png)

课程在 7 月 5 日正式开始。但我很清楚一件事：**如果就这么裸着进去听课，大概率会被链、合约、钱包、Gas 这些专业术语直接打懵**。就像第一次进健身房，还没搞清楚器械怎么用就被拉去上高强度团课——不是不能练，是练完一身伤。所以我提前做了一点准备。而这篇文章就是在讲这件事：**在正式进入 Web3 之前，我是怎么用 Web2 的视角去补齐认知的。**

## Web2：我所习惯的那一套世界

如果用一句话总结 Web2 的开发方式，那就是：

> **一切都围绕服务器展开。**

在 Web2 里，一个系统通常是非常标准化的分层结构：

- 前端负责 UI 和交互
- 后端负责业务逻辑
- 数据库存数据
- API 作为连接层

在实际开发中，无论是写 React 还是写 Node，本质上都是在这个结构里做“填空题”——你只需要把各层之间的逻辑补完整，系统就能运行。从这个角度看，Web2 的核心其实在后端。前端是入口，后端才是系统真正的执行中心。可以用一个更直观的类比理解：

> 前端 = 门店 + 菜单（用户能看到的部分）
> 后端 = 厨房 + 仓库（真正执行与存储的地方）

用户进入系统时，接触到的是界面和交互，但真正的数据处理与业务逻辑，都发生在后端。

### Web2 的另一个关键特征：抽象与不可见性

在 Web2 中，对于普通用户来说，后端是不可见的。用户只会看到：

- 页面
- 功能
- 交互结果

但不会接触到：

- 数据存储方式
- 业务逻辑实现
- 权限控制细节

因此从用户视角来看，系统的内部实现是被高度抽象的。可以说 Web2 的一个核心特征是：**系统内部是封闭的实现细节，外部只提供可用结果。**

此外 Web2 的核心结构是中心化的：数据由平台统一存储、规则由后端控制、权限由系统分配。导致了用户的浏览记录、社交关系、发布内容，本质上都由平台持有与管理。这带来一个很重要的结构特征：那就是开发者和用户都在使用系统，但系统本身由平台作为中心进行控制与调度。

- 用户是使用者
- 开发者是构建者
- 平台是控制者

三者角色是分离的，而系统运行依赖于这个中心结构。Web2 让用户“参与”了互联网，但没让用户“拥有”互联网

## Web3：我在进入前做的第一轮理解

其实 Web3 这个名词我在几年前就听说过了，关于 Web3 最常用的描述就是“去中心化”。同时我对 Web3 的了解也仅仅局限在虚拟货币、区块链、智能合约这些词汇上。在我理解中，Web3 可以先粗略拆成一个四层结构：

- 钱包：用户入口（身份与签名）
- 区块链：状态存储层（所有数据最终记录在这里）
- 智能合约：规则执行层（系统逻辑）
- 前端应用：交互入口（访问链上系统）

如果用一句话概括：

> Web3 是一个“由钱包驱动、由合约执行、由链维护状态”的系统。

### 区块链：

假设有这样一个虚拟的小镇，镇上一共有一百户居民。在这个镇子上有个公告栏，所有人都可以往上面贴纸条记录或者公布一些事情。不过想要张贴纸条需要满足以下规则：

- 每一张纸条必须按顺序编号（链式结构）
- 想要张贴新纸条必须得到全镇一半以上的居民同意（共识机制）
- 纸条具有神奇的魔力，一旦张贴就撕不下来同时也修改不了（不可篡改）

这三条规则共同构成了区块链，它的价值不在于技术多复杂而是解决了人与人的信任危机。大家不需要一个权威机构或者服务器提供商来充当公证人，而是共同利用这一套公开的规则来维持整个小镇的秩序

### 智能合约：

智能合约的概念也不难理解，它本质上是一个自动触发的纸条。还是在这个小镇上，我将一张纸条贴在了公告栏上。这张纸条同时满足了按照顺序、由小镇一半以上的人做公证、我贴上去后内容不能被修改的条件。纸条的内容是：

> “如果明天下午3点，张三把电动车送到我家楼下，那我的账户就自动转给他2000块。”

而且这个纸条不只是一个普通的纸，它是一个高科技纸条，一旦检测到前项被满足就会自动执行后项。接着第二天，张三把电动车送过来了，然后纸条上的开关被触发，紧接着我的账户自动将钱打给了张三。而这一切都是智能合约的作用。

再具体一点，可以把**智能合约理解为自动贩卖机：投币（满足条件）→ 出货（自动执行）**。

### 加密钱包：

加密钱包虽然被称为钱包，但实际并不是存钱的，它更是一把钥匙。在整个小镇上存在好多个一模一样的保险箱，每一个保险箱都存储了小镇居民的资产。同时每一个小镇居民都有一把属于自己的钥匙。同时在钥匙和保险箱上都会有一串编号，方便每一位小镇居民可以快速找到自己的保险箱。这个就是**公钥**，公钥可以公开展示给每一个人，作用是方便快速找到你的保险箱。与之对应的还有**私钥**，私钥则是独属于你自己的秘密，作用是打开保险箱。而私钥一旦丢失，那么所有人都可以将你的保险箱搬空。（其实我感觉还很像霍格沃兹中的古灵阁）

在 Web3 中，钱包的本质不是“存钱工具”，而是一个用来生成身份签名的工具。Web3 上的所有操作（转账、交互合约）都不是登录系统完成的，而是通过“签名”证明你对某个地址的控制权。

### 前端应用（DApp）：

Web3 的前端应用也被称为 DApp，就是去中心化App。相比于传统的 App，DApp“舍弃”了后端，而是把后端的核心（数据存储 + 业务逻辑）外包给了区块链。用户通过打开前端界面，接着连接自己的钱包，随后 DApp 会直接访问对应的区块链。

DApp 更像是一台电视机，输入哪个信号源（区块链），就显示什么内容。哪怕这台电视机坏掉，换一台照样可以继续观看。网站只是个“显示器”，真正的数据和信息都存储在更安全的区块链上。相比于传统的应用，一旦服务器挂掉或者被平台封号你的数据就会全部消失，但是 DApp 的话只需要换个网址，连上钱包，一模一样继续用，因为数据不在网站里，在区块链上。

## Web2 vs Web3：我理解中的核心差异

如果把 Web2 和 Web3 放在同一个视角下对比，其实可以很清晰地看到两者的结构差异：

> Web2 本质上是一个“中心化系统”：

- 数据由平台集中存储
- 业务逻辑运行在后端服务器
- 用户通过前端访问服务
- 信任来源是平台本身

> 而 Web3 则是一个“去中心化状态系统”：

- 数据记录在链上
- 规则通过智能合约公开执行
- 用户通过钱包直接参与系统
- 信任来源是规则与共识

在实际理解过程中，我逐渐意识到两者真正的区别不在于“用了什么技术”，而在于默认前提完全不同：

- Web2 默认的是：“系统一定存在一个中心来维护数据与规则”

- Web3 默认的是：“系统可以不存在中心，规则本身就是系统”

这个差异会直接影响你如何设计和理解整个应用结构。从 Web2 到 Web3 的过程中，我的认知其实发生了三个比较明显的转变：

### 从“应用思维”到“状态思维”

- Web2 中我们更多地去关注：功能怎么实现？接口怎么设计？后端怎么写？

- Web3 中，问题变成：状态是谁维护的？状态如何变化？谁有权限改变状态？

### 从“后端中心”到“链上中心”

- Web2 中后端是系统核心，前端只是入口

- Web3 中链才是系统核心，前端只是交互工具而钱包成为唯一“身份入口”

### 从“信任平台”到“信任规则”

- Web2 中我们信任平台不会出错

- Web3 中我们信任代码和链上的规则不会被随意篡改

## 结尾：用已有系统去理解新系统

在正式进入 Monad Builder Camp 的学习之前，我做的这些准备，本质上不是为了掌握 Web3，而是为了让自己在面对新体系时不至于从零开始理解。至少现在，我可以在看到“合约 / 钱包 / 链上交互”这些概念时，不再把它们当作孤立名词，而是放回到一个完整的系统结构中去理解。学习新技术的关键，不是从零理解，而是建立“映射关系”。

真正重要的是：

> 能不能快速找到新旧系统之间的对应关系
> 能不能用已有经验降低理解成本
> 能不能把“陌生概念”转成“结构问题”
